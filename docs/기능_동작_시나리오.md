# 기능 동작 시나리오

## 1) 로그인/인증 유지

1. 사용자 로그인: `POST /api/v1/auth/login`
2. 응답으로 access token 수신 + refresh token 쿠키 설정
3. 프론트는 API 호출 시 `Authorization: Bearer <access_token>` + `credentials: include`
4. 401 발생 시 `GET /api/v1/auth/token/refresh`로 access token 재발급 후 재시도

근거:

- `frontend/src/App.jsx`
- `app/apis/v1/auth_routers.py`

## 2) 보호자-복약자 연동

1. 복약자(또는 초대권한 사용자)가 초대코드 생성: `POST /api/v1/users/invite-code`
2. 보호자가 초대코드로 연동: `POST /api/v1/users/link`
3. 연동 목록 확인: `GET /api/v1/users/links`
4. 필요 시 연동 해제: `DELETE /api/v1/users/links/{link_id}`

근거:

- `app/apis/v1/invitation_routers.py`
- `app/services/invitation.py`
- `app/tests/user_apis/test_invitation_apis.py`

## 3) 문서 업로드 -> OCR -> 추출약 확정

1. 문서 업로드: `POST /api/v1/documents/upload`
2. OCR 비동기 작업 생성 및 처리
3. 상태 조회: `GET /api/v1/documents/{document_id}/status`
4. 추출약 조회: `GET /api/v1/documents/{document_id}/drugs`
5. 사용자 검수/확정: `PATCH /api/v1/documents/{document_id}/drugs` (`confirm=true`)
6. 확정 시 `patient_meds` 및 관련 스케줄 동기화

근거:

- `app/services/ocr.py`
- `app/services/documents.py`
- `app/apis/v1/document_routers.py`

## 4) 문서 삭제 시 복약체크 영향 제거

동작 의도:

- OCR 문서가 삭제되면 그 문서 기반 복약정보가 복약체크에 남지 않아야 함

현재 구현:

1. 문서 삭제: `DELETE /api/v1/documents/{document_id}`
2. 트랜잭션 내 처리
- 해당 문서의 active `patient_meds` 비활성화
- 연결된 active `med_schedules`를 `inactive`로 변경
- 연결된 `med_schedule_times`를 `is_active=false` 처리
- 문서 상태 `deleted` + `deleted_at` 기록
3. 복약현황 조회는 active 스케줄만 집계하므로 삭제 문서 기반 일정이 제외됨

근거:

- `app/services/documents.py` (`soft_delete_document`)
- `app/repositories/med_schedule_repository.py` (`list_active_schedules_by_patient_in_range`)
- `app/tests/user_apis/test_document_drugs_apis.py` (삭제 동기화 테스트)

## 5) 복약체크 상태 처리

1. 체크: `POST /api/v1/schedules/{schedule_id}/check`
2. 취소: `DELETE /api/v1/schedules/{schedule_id}/check`
3. 건너뛰기: `POST /api/v1/schedules/{schedule_id}/skip`
4. 상태조회: `GET /api/v1/schedules/status?patient_id=&from=&to=`
5. 이행률: `GET /api/v1/schedules/adherence?patient_id=&from=&to=`

근거:

- `app/apis/v1/medication_intake_api.py`
- `app/services/medication_intake_service.py`

## 6) 보호자/복약자 모드 전환

현재 프론트 동작:

1. 로그인 역할 + 연동 상태를 조합해 mode 옵션 계산
2. 모드 전환 시 `login_role` 저장값 업데이트
3. 관리자 대시보드 경로 예외 처리(`/dashboard`)
4. 각 페이지는 현재 모드(`PATIENT`/`CAREGIVER`/`ADMIN`) 기준으로 데이터 요청 대상을 변경

근거:

- `frontend/src/App.jsx`
- `frontend/src/components/AppLayout.jsx`
- `frontend/src/MedicationCheckPage.jsx`

## 7) 알림/리마인드

1. 알림 목록 조회 및 읽음/삭제/전체읽음/개수 관리
2. 보호자가 환자에게 수동 리마인드 발송: `POST /api/v1/notifications/remind`
3. 알림 설정 조회/수정 가능

근거:

- `app/apis/v1/notification_routers.py`
- `app/services/notifications.py`
- `frontend/src/NotificationPage.jsx`
