# 서비스와 모델 관계

## 1) 도메인별 서비스

| 도메인 | 주요 Service | 주요 Router | 핵심 모델 |
|---|---|---|---|
| 인증 | `AuthService`, `JwtService` | `auth_routers.py` | `User`, `AuthAccount`, `UserRole` |
| 사용자/연동 | `InvitationService`, `UserManageService`, `patient_profile_service` | `invitation_routers.py`, `user_routers.py`, `patient_profile_routers.py` | `User`, `Patient`, `CaregiverPatientLink` |
| 문서/OCR | `DocumentService`, `OcrService`, `BarcodeService` | `document_routers.py` | `Document`, `OcrJob`, `ExtractedMed`, `PatientMed` |
| 복약체크 | `MedicationIntakeService` | `medication_intake_api.py` | `MedSchedule`, `MedScheduleTime`, `IntakeLog` |
| 일정 | `HospitalCalendarService` | `hospital_calendar_api.py` | `HospitalSchedule` |
| 알림 | `NotificationService` | `notification_routers.py` | `Notification`, `NotificationSettings` |
| AI 가이드 | `GuideService` | `guide_routers.py` | `Guide`, `GuideGenerationJob`(서비스 내부 참조) |
| AI 채팅 | `ChatService` | `chat_routers.py` | `ChatSession`, `ChatMessage`, `ChatFeedback` |
| 관리자 | `DashboardService` | `dashboard_routers.py` | 통계 집계 모델 다수 |

## 2) 핵심 데이터 흐름

문서 기반 복약 데이터 흐름:

1. `Document` 업로드
2. `OcrJob` 생성/처리
3. `ExtractedMed` 생성
4. 사용자 확정 시 `PatientMed` 동기화
5. `PatientMed` 기반 `MedSchedule`/`MedScheduleTime` 사용
6. 복약체크 결과는 intake 로그로 저장 후 상태 계산

삭제 동기화 흐름:

1. `Document.status = deleted`
2. 해당 문서 소스의 `PatientMed.is_active = false`
3. 관련 `MedSchedule.status = inactive`
4. 관련 `MedScheduleTime.is_active = false`

## 3) 권한 체크 공통 패턴

- `get_request_user` 의존성으로 요청 사용자 식별
- 보호자 접근은 `CaregiverPatientLink(status=active)` 기반 검증
- 가이드/채팅은 환자/세션 접근 가능 여부를 별도 함수로 추가 검증

## 4) 프론트-백엔드 결합 포인트

- 프론트 API prefix는 고정: `"/api/v1"`
- 정적 앱 경로 base: `"/auth-demo/app/"`
- 인증 유지: access token(local storage) + refresh token(cookie)
