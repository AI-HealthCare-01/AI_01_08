# 배포/운영 가이드

## 1) 현재 배포 구조 (권장 기본안)

현재 프로젝트는 **통합 배포**를 전제로 구성되어 있습니다.

- Frontend(Vite build 결과물)를 `app/ui/auth-demo`에 생성
- FastAPI + Nginx가 `/auth-demo/app` 경로로 정적 서빙
- API는 same-origin `/api/v1`로 호출

관련 설정:

- `frontend/vite.config.js`
- `app/main.py`
- `nginx/default.conf`
- `docker-compose.prod.yml`

## 2) 로컬 실행

1. 환경파일 준비
- `cp envs/example.local.env envs/.local.env`

2. 전체 스택 실행
- `docker-compose up -d --build`

3. 접속 확인
- API docs: `http://localhost/api/docs`
- 앱: `http://localhost/auth-demo/app`
- 문서: `http://localhost/api/project-docs/`

## 3) 프론트 빌드/반영

프론트는 별도 dist 배포가 아니라 앱 디렉토리로 출력됩니다.

- `cd frontend`
- `npm install`
- `npm run build`

출력:

- `app/ui/auth-demo/index.html`
- `app/ui/auth-demo/assets/*`

## 4) 운영 배포 체크리스트

사전 점검:

- `.env` 값 점검 (`DB_*`, `COOKIE_DOMAIN`, OAuth, OCR, MFDS)
- `CORS_ALLOW_ORIGINS` 운영 도메인 반영
- `COOKIE_DOMAIN` 실제 도메인 반영
- 프론트 빌드 최신 산출물 반영

배포 후 점검:

- 로그인/토큰 재발급 동작
- 모드 전환(복약자/보호자/관리자)
- 문서 업로드/OCR/추출약 확정
- 문서 삭제 후 복약체크 반영 제거
- 알림 조회/리마인드 발송

## 5) Vercel 분리 배포 시 주의

현재 코드는 `/api/v1` 상대경로 + 쿠키 포함 인증을 전제로 합니다.

분리 배포를 하려면 최소 아래를 함께 변경해야 합니다.

- 프론트 API base URL 분리(환경변수화)
- 백엔드 CORS에 프론트 도메인 추가
- 쿠키 정책(`domain`, `secure`, `sameSite`) 교차 도메인 정책 점검
- `/auth-demo/app` base path 전략 재설계 (루트 또는 rewrites)

## 6) 장애 대응 기본 절차

1. 증상 재현 API 확인 (`/api/docs`로 요청 스펙 점검)
2. DB 레코드 상태 확인 (document/schedule/intake/notification)
3. OCR job 상태/오류코드 확인
4. Nginx 라우팅 및 정적 파일 반영 여부 확인
5. 필요 시 직전 정상 이미지/태그로 롤백
